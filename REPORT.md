# AI для тоғызқұмалақ — Технический отчёт

## 1. Цель проекта

Разработка AI-движка для тоғызқұмалақ (togyz kumalak) — казахской настольной игры семейства манкала. Цель — движок чемпионского уровня, способный конкурировать с сильнейшими игроками.

### Краткие правила
- Доска: 2 стороны × 9 отау (лунок), по 9 камней в каждой. Всего 162 камня.
- Ходы: выбрать свой отау с камнями и распределить их по одному по следующим отау.
- Захват: если последний камень попадает на сторону противника и создаёт чётное количество — камни забираются в қазан (хранилище).
- Тұздық: если последний камень попадает в отау противника и создаёт ровно 3 — этот отау становится тұздық (ловушка), навсегда собирающая проходящие камни. Ограничения: нельзя на 9-й отау, каждый может иметь только один тұздық, нельзя на тот же номер что у противника.
- Победа: кто первый собрал ≥ 82 камня в қазан. Если у одного игрока не осталось камней на доске — оставшиеся считаются.

## 2. Архитектура движка

Движок написан на **Rust**. Основные модули:

| Модуль | Файл | Назначение |
|--------|------|------------|
| Board | `board.rs` | Представление доски, правила, генерация ходов |
| Search | `search.rs` | Alpha-Beta поиск с итеративным углублением |
| Eval (HCE) | `eval.rs` | Ручная оценочная функция (Texel-tuned) |
| NNUE | `nnue.rs` | Нейросетевая оценка (квантизованный inference) |
| TT | `tt.rs` | Таблица транспозиций |
| Zobrist | `zobrist.rs` | Zobrist-хэширование позиций |
| Texel | `texel.rs` | Автоматическая настройка весов оценки |
| Datagen | `datagen.rs` | Генерация обучающих данных из самоигры |
| Main | `main.rs` | CLI-интерфейс, serve-протокол для веб-сервера |

### 2.1. Поиск (Search)

**Алгоритм**: Negamax Alpha-Beta с fail-soft + Iterative Deepening (до depth 64).

Реализованные техники:

| Техника | Детали |
|---------|--------|
| **PVS** (Principal Variation Search) | Null-window re-search для ходов после первого |
| **Null Move Pruning** | R=2, включается при depth ≥ 3, наших камней > 10, не в эндшпиле |
| **LMR** (Late Move Reductions) | После 3-го хода, depth ≥ 3, не в эндшпиле. Reduced search depth-2, re-search при улучшении |
| **Killer Moves** | 2 слота на каждый ply. Бонус: 1-й +9000, 2-й +8000 к сортировке |
| **History Heuristic** | history[side][pit] += depth² при beta-cutoff |
| **Transposition Table** | 64 MB, always-replace, Zobrist-хэш |
| **Quiescence Search** | Только захватывающие ходы (landing на чётное число) |
| **Repetition Detection** | Двухуровневая: по истории игры + по пути в дереве поиска |
| **Endgame Extensions** | +1 ply при ≤ 15 камней на доске (ply ≤ 3) или ≤ 30 камней (ply = 0) |

**Сортировка ходов** (по приоритету):
1. Ход из TT: +100,000
2. Захватывающий ход: +10,000
3. Killer #1: +9,000 / Killer #2: +8,000
4. History heuristic
5. Количество камней в отау

**Управление временем**: проверка каждые 4096 нод. Настраиваемый лимит (по умолчанию 3–5 секунд).

### 2.2. Zobrist-хэширование

- Генератор: xorshift64 с seed `0x12345678_DEADBEEF`
- Компоненты хэша:
  - Отау: 2 стороны × 9 отау × 33 bucket'а (камни 0–32+) = 594 ключа
  - Қазан: 2 стороны × 163 значения = 326 ключей
  - Тұздық: 2 стороны × 10 вариантов = 20 ключей
  - Сторона хода: 1 ключ
- Квантизация камней: 0–31 отдельные ключи, 32+ — один общий

### 2.3. Таблица транспозиций

- Размер: 64 MB (настраиваемый)
- Запись: `{ hash: u64, depth: i32, score: i32, flag: Exact|Lower|Upper, best_move: i8 }`
- Стратегия замещения: Always Replace
- Индексация: `hash & mask` (степень двойки)

## 3. HCE — Hand-Crafted Evaluation

Ручная оценочная функция, настроенная методом **Texel Tuning** на **77,000 позициях** с PlayOK (реальные партии). Результат: **44% снижение ошибки** по сравнению с начальными весами.

### 3.1. Факторы оценки

| # | Фактор | Вес | Описание |
|---|--------|-----|----------|
| 1 | Material (қазан разница) | 21 | Основной фактор: (мой_қазан − чужой_қазан) × 21 |
| 2 | Endgame material boost | −6 | Дополнительно при total_kazan > 100 |
| 3 | Proximity to win | 30 | Бонус за каждый камень свыше 70 в қазане |
| 4 | Тұздық base | 504 | Наличие тұздық — большой бонус |
| 5 | Тұздық center bonus | 63 | Центральные позиции (3–5) получают ×3 бонус |
| 6 | Тұздық threat | 11 | Каждый отау противника с 2 камнями (угроза тұздық) |
| 7 | Pit stones | 3 | Разница камней на доске (не в қазане) |
| 8 | **Mobility** | **124** | **Разница доступных ходов — самый важный позиционный фактор** |
| 9 | Empty pit penalty | −5 | Штраф за каждый пустой отау |
| 10 | Large pit bonus | 1 | Бонус за камни сверх 10 в отау |
| 11 | Capture opportunities | 4 | Возможности захвата (посадка на чётное у противника) |
| 12 | **Starvation pressure** | **15** | Квадратичный бонус при камнях противника ≤ 9 |
| 13 | **Finishing bonus** | **200** | Добивание: material_diff > 5 и у противника ≤ 3 камня |

### 3.2. Формула starvation

```
Если камни противника на доске ≤ 9:
    bonus = (10 − камни_противника)² × 15

Если material_diff > 5 И камни противника ≤ 3:
    bonus += (4 − камни_противника) × 200
```

| Камни противника | Starvation | Finishing | Итого |
|------------------|-----------|-----------|-------|
| 1 | 81 × 15 = 1215 | 3 × 200 = 600 | **1815** |
| 3 | 49 × 15 = 735 | 1 × 200 = 200 | **935** |
| 5 | 25 × 15 = 375 | — | **375** |
| 9 | 1 × 15 = 15 | — | **15** |

### 3.3. Методология Texel Tuning

1. Загрузка 77K позиций из партий на PlayOK (CSV: отау, қазан, тұздық, результат)
2. Метрика: MSE = `mean((result − sigmoid(eval / K))²)`
3. Поиск оптимального K: грубый (шаг 50 от 50 до 1000) → точный (±50, шаг 5). **Найденный K = 1050.**
4. Локальный поиск по каждому весу с уменьшающимися шагами: [50, 20, 10, 5, 2, 1]
5. 11 настраиваемых параметров, итерация до сходимости

## 4. NNUE — нейросетевая оценка

### 4.1. Архитектура

```
Input(40) → Linear(256) → ClippedReLU → Linear(32) → ClippedReLU → Linear(1)
```

- Параметров: **18,753**
- Размер файла весов: **37,510 байт**
- Квантизация: `float × 64 → int16`
- Активация: `ClippedReLU(x) = clamp(x, 0, SCALE)`, где SCALE = 64

### 4.2. Входные признаки (40)

| Индексы | Признак | Нормализация |
|---------|---------|--------------|
| 0–8 | Отау текущего игрока | / 50.0 |
| 9–17 | Отау противника | / 50.0 |
| 18 | Қазан текущего игрока | / 82.0 |
| 19 | Қазан противника | / 82.0 |
| 20–29 | Тұздық текущего (one-hot, 10 слотов: 0–8 = позиция, 9 = нет) | 0 или 1 |
| 30–39 | Тұздық противника (one-hot) | 0 или 1 |

Все признаки — относительно стороны хода (side-to-move perspective).

### 4.3. Гибридная формула NNUE + HCE

Смешивание зависит от количества камней на доске (не в қазанах):

| Камни на доске | NNUE | HCE |
|----------------|------|-----|
| > 30 (дебют/миттельшпиль) | 100% | 0% |
| 15–30 (эндшпиль) | 50% | 50% |
| ≤ 15 (глубокий эндшпиль) | 25% | 75% |

**Формула в коде** (`search.rs`):
```rust
if total_board_stones <= 15 {
    (nnue_score + hce_score * 3) / 4
} else if total_board_stones <= 30 {
    (nnue_score + hce_score) / 2
} else {
    nnue_score
}
```

Обоснование: NNUE хорошо распознаёт паттерны в дебюте/миттельшпиле, но HCE имеет Texel-tuned знание эндшпиля (starvation, finishing), которого NNUE не хватает.

## 5. Генерация обучающих данных

### 5.1. Self-play datagen (`datagen.rs`)

- **Многопоточность**: настраиваемое число потоков (по умолчанию — все доступные ядра)
- **Поиск**: Alpha-Beta с TT (4 MB на поток)
- **Разнообразие**: первые 8 ходов — случайные (для разнообразия дебютов)
- **Adjudication**: если оценка > ±8500 в течение 4 ходов подряд — досрочное окончание
- **Лимит**: максимум 300 полуходов на партию
- **Калибровка**: автоматическая нормализация оценок до средней абсолютной ≈ 500

### 5.2. Формат записи (26 байт на позицию)

```
bytes 0–8:   pits_white[9]   (u8)
bytes 9–17:  pits_black[9]   (u8)
byte  18:    kazan_white     (u8)
byte  19:    kazan_black     (u8)
byte  20:    tuzdyk_white    (i8, −1 = нет)
byte  21:    tuzdyk_black    (i8, −1 = нет)
byte  22:    side_to_move    (u8, 0 = white)
bytes 23–24: eval            (i16 LE)
byte  25:    result          (u8, 0 = white_loss, 1 = draw, 2 = white_win)
```

### 5.3. Дополнительные источники данных

| Источник | Описание | Позиций |
|----------|----------|---------|
| Self-play HCE (V1) | Самоигра depth 8 с HCE-оценкой | 1.24M |
| Self-play HCE (Mac) | Дополнительная генерация | 2.46M |
| Combined HCE | V1 + Mac | 3.71M |
| V7 improved self-play | С repetition detection и NNUE-гибридом | 3.37M |
| Master games (Elo ≥ 1800) | Конвертация из базы партий PlayOK | ~100K |
| Master all-Elo | Все партии с базы | ~200K |
| Synthetic endgame | Случайные глубокие эндшпили + near-win | ~800K |

### 5.4. Подготовка данных для обучения

Скрипты подготовки (`prepare_v9_data.py`, `prepare_v9b_data.py`) объединяют источники с **дупликацией эндшпильных позиций** (2–3×) для акцента на критических позициях.

## 6. Обучение NNUE

### 6.1. Функция потерь

```
pred_wp = sigmoid(prediction / (K / 4))        # K = 1050
eval_wp = sigmoid(eval / (K / 4))
target  = λ × eval_wp + (1 − λ) × game_result  # λ = 0.75
loss    = mean((pred_wp − target)² × sample_weights)
```

- **λ (lambda)**: баланс между оценкой движка и результатом партии
- **Sample weights**: эндшпильные позиции (≤ 30 камней) — вес 2×, глубокие (≤ 15) — вес 3×
- **Adaptive lambda**: для позиций без оценки (eval = 0) используется λ = 0 (чистый результат)

### 6.2. Оптимальные гиперпараметры

| Параметр | Значение | Примечание |
|----------|----------|------------|
| Batch size | **4096** | Критически важно — batch=8192 теряет ~258 Elo |
| Learning rate | 0.001 | Adam |
| Weight decay | 1e-5 | L2-регуляризация |
| Scheduler | CosineAnnealingLR | LR → 0 к концу обучения |
| Epochs | 100 | Диапазон 75–150 |
| Lambda | 0.75 | 75% оценка + 25% результат |
| Validation | 10% данных, max 50K | Hold-out для раннего сохранения лучших весов |

### 6.3. Экспорт весов

- **JSON** (`nnue_weights.json`): полная точность, для отладки
- **Binary** (`nnue_weights.bin`): квантизация float → int16 (×64), 37,510 байт. Загружается в Rust-движок.

### 7.5. Ключевые выводы

#### 1. Batch size критичен (258 Elo разницы)
```
V1  (batch=4096, 100 эпох): 79% = +230 Elo
ExpM (batch=8192, 100 эпох): 46% = −28 Elo
```
Те же данные, тот же λ, те же эпохи. Только batch size. Большие батчи → более острые минимумы → хуже обобщение.

#### 2. Больше данных ≠ лучше
```
V1 (1.24M):  79% = +230 Elo
V4 (3.7M):   72% = +164 Elo
V9 (10.9M):  слабее V7
V9b (11.37M): слабее V7
```
Дополнительные данные размывали качество. Особенно синтетические эндшпили и избыточная дупликация эндшпильных позиций сдвигали распределение обучающих данных, и сеть теряла дебютное/миттельшпильное знание.

#### 3. λ = 0.75 оптимально (300 Elo разницы)
```
V1 (λ=0.75): 79% = +230 Elo
V5 (λ=1.0):  ~40% = ~−70 Elo
```
Чисто оценочный таргет (без учёта результата партии) — значительно хуже. Результат партии даёт критически важный сигнал.

#### 4. Val loss не предсказывает силу игры
```
Exp H: val loss = 0.0456 (лучший!) → 25% в матче (~−200 Elo)
Exp M: val loss = 0.0679            → 46% в матче
V1:    val loss = неизвестен        → 79% в матче (+230 Elo)
```
**Вывод**: оценивать модели только через матчи (минимум 50 партий).

#### 5. Гибрид NNUE + HCE — главный прорыв
Чистый NNUE V7 набирает лишь ~45% против HCE. Но в гибридном режиме (NNUE в дебюте, HCE в эндшпиле) — **83.8% (+285 Elo)**. NNUE хорошо распознаёт позиционные паттерны, HCE надёжно считает эндшпильную математику.

#### 6. PlayOK данные бесполезны для NNUE
Все эксперименты с PlayOK-позициями давали val loss ~0.18–0.19 (в 4× хуже, чем HCE-данные: ~0.045). Вероятная причина: оценки HCE-движка не соответствуют сложным позициям из мастерских партий.

#### 7. Синтетические эндшпильные данные вредят
V9 (с synthetic endgame) и V9b (без synthetic, но с 3× дупликацией) — оба хуже V7. Искусственные эндшпильные позиции имеют нереалистичное распределение, а чрезмерная дупликация «затапливает» данные.

## 8. Инфраструктура

### 8.1. Persistent Engine Protocol (serve-режим)

**Проблема**: ранее каждый ход запускал новый процесс → TT терялась, история терялась.

**Решение**: постоянный процесс, протокол через stdin/stdout:

| Команда | Действие |
|---------|----------|
| `newgame` | Сброс TT, killers, history → `ready` |
| `position <pos>` | Добавить хэш позиции в историю → `ready` |
| `go time <ms> pos <pos>` | Поиск лучшего хода → `bestmove ...` |
| `ping` | Проверка → `pong` |
| `quit` | Завершение |

**Эффект**: TT сохраняется между ходами, что даёт ≈ 50–100 Elo дополнительно.

### 8.2. Web-сервер (`web/server.py`)

- **Flask** с persistent subprocess движка
- **Thread-safe**: `engine_lock` для доступа к подпроцессу
- Автоматический перезапуск при падении движка
- 4 API-эндпоинта: `/` (UI), `/api/move`, `/api/newgame`, `/api/position`

### 8.3. Дебютная книга

- **872 позиции** из **1755 мастер-партий** (оба игрока Elo ≥ 1600)
- Глубина: до 16 полуходов
- Формат: JSON с частотой ходов и статистикой побед/ничьих/поражений
- Выбор хода: взвешенный рандом с весом = `count × (0.3 + win_rate)`
- Фильтрация: минимум 3 партии на позицию, минимум 2 партии на ход

### 8.4. Web-интерфейс (`web/index.html`)

- **Лобби**: выбор стороны, контроль времени (1/3/5/10 сек), режим (игра/анализ)
- **Доска**: визуализация 18 отау + 2 қазана, маркеры тұздық, анимация раздачи камней
- **Боковая панель**: история ходов (кликабельная в режиме анализа), панель анализа движка (лучший ход, оценка, глубина, NPS)
- **Интерфейс полностью на казахском языке**

## 9. Текущий уровень и слабые стороны

### 9.1. Оценка силы

| Уровень оппонента | Результат |
|-------------------|-----------|
| Любители | Уверенно побеждает |
| Опытные (1600–1900) | Конкурентоспособен, ~50–60% побед |
| **Чемпионы** | **Недостаточно** |

### 9.2. Конкретные слабости

1. **Эндшпильная стратегия** — starvation-логика добавлена, но движок не имеет глубокого позиционного понимания эндшпильных планов. Нет endgame tablebases.

2. **Тұздық стратегия** — оценка тұздық статическая (фиксированные бонусы). Нет динамического планирования: «через 3 хода я могу создать тұздық на 4-м отау». Нет комбинационного расчёта тұздық.

3. **Качество NNUE-данных** — обучение на самоигре (self-play) HCE-движка. Сеть учится копировать HCE, а не превосходить его. Нет данных от сильных игроков с корректными оценками.

4. **Глубина поиска** — за 3 секунды достигает depth 28–30. Для чемпионского уровня может потребоваться 35+.

5. **Дебютная книга** — 872 позиции (16 полуходов). Мало для полного покрытия дебютной теории.

6. **Нет self-play loop** — движок не проходил циклы «сгенерировать данные → обучить → сгенерировать данные лучшим движком → обучить снова».

## 11. Вопросы для экспертов

### По обучению NNUE

1. **Self-play loop**: как правильно организовать цикл усиления? Сколько итераций, как оценивать прогресс между поколениями?

2. **Архитектура сети**: текущая 40→256→32→1 (18K параметров) — стоит ли масштабировать? До какого размера имеет смысл для игры с branching factor 4–9?

3. **Данные из мастер-партий**: как правильно использовать? Наши эксперименты с PlayOK-данными провалились (val loss 4× хуже). Проблема в оценках или в позициях?

4. **Batch size аномалия**: 4096 >> 8192 по силе игры (258 Elo). Это нормально для таких маленьких сетей? Стоит ли пробовать ещё меньше (2048, 1024)?

### По поиску

5. **Aspiration windows**: стоит ли добавлять? Какие начальные окна для тоғызқұмалақ?

6. **Singular extensions**: насколько они эффективны при таком маленьком branching factor?

7. **Multi-PV**: нужен ли для анализа? Текущий движок показывает только 1 лучший ход.

### По оценке

8. **Тұздық комбинации**: как оценивать потенциал создания тұздық за несколько ходов? Это задача оценки или поиска?

9. **Endgame tablebases**: имеет ли смысл для тоғызқұмалақ? State space с тұздық довольно большой.

10. **Tempo**: как оценивать инициативу в тоғызқұмалақ? Текущая оценка не учитывает, кто «нападает».

### По общей стратегии

11. **Что даст больший прирост**: улучшение поиска (глубже/шире) или улучшение оценки (точнее)?

12. **Benchmarking**: как объективно измерить силу без рейтинговой системы? Текущий метод — матчи HCE vs NNUE-hybrid.
